# 知识库日历功能 PRD

## 1. 产品概述

### 1.1 功能名称

知识详情页「添加到日历」与独立 Calendar 页面。

### 1.2 背景

知识库中的微信公众号文章、网页、GitHub 项目、GPT 对话、抖音视频等内容，可能包含未来的重要日期，例如：

- 报名截止日期
- 论文或项目提交日期
- 产品发布时间
- 活动举办日期
- 服务到期日期
- 课程作业截止日期
- 优惠或免费额度结束日期

目前用户在阅读内容时，需要手动记住或另外记录这些日期，容易遗漏，也难以从日期返回原始知识内容。

因此，需要在知识详情页加入「添加到日历」功能，并建立独立的 Calendar 页面，统一管理与知识内容相关的日期事项和用户手动创建的日程。

---

## 2. 产品目标

### 2.1 核心目标

1. 在知识详情页中提供清晰的「添加到日历」入口。
2. 系统从知识内容中快速识别可能的重要日期，并提供推荐日期。
3. 用户可以接受推荐日期，也可以自行选择其他日期。
4. 日历事项的时间颗粒度只到日期，不涉及具体小时和分钟。
5. 日历作为独立一级页面存在，不放入 Dashboard。
6. 从知识详情页创建的日历事项，需要保留与原知识条目的关联。
7. 用户可以在 Calendar 页面新增、编辑和删除日历事项。

### 2.2 成功标准

用户能够在知识详情页中，通过不超过三步完成日历事项创建：

1. 点击「添加到日历」。
2. 确认或修改系统推荐的日期。
3. 点击「添加」。

创建完成后，该事项能够在 Calendar 页面中正确显示，并能够返回原知识详情页。

---

## 3. 非目标

当前版本不包含以下功能：

- 小时或分钟级时间安排
- 日视图或小时级周视图
- 拖拽修改日期
- Google Calendar、Outlook 等外部日历同步
- 浏览器、邮件或手机推送提醒
- 重复日程
- 多人共享日历
- AI 自动安排用户的空闲时间
- 自动将所有识别日期直接加入日历
- Dashboard 内嵌完整日历
- 根据用户空闲时间自动选择日期

---

## 4. 目标用户与使用场景

### 4.1 目标用户

需要同时管理大量知识内容和未来事项的个人用户，例如：

- 学生
- 研究人员
- 开发者
- 求职者
- 内容收藏者
- 个人知识库使用者

### 4.2 典型使用场景

#### 场景一：文章包含明确截止日期

用户收藏了一篇文章，正文中写有：

> 报名将在 2026 年 8 月 1 日截止。

用户进入详情页，点击「添加到日历」。

系统推荐：

- 事项名称：报名截止
- 日期：2026-08-01

用户确认后添加。

#### 场景二：文章包含多个日期

正文中写有：

> 项目将在 7 月 20 日开放报名，并于 8 月 1 日截止。

系统识别到两个日期，但默认推荐截止日期：

- 推荐日期：2026-08-01
- 推荐原因：与“截止”关键词关联

用户仍然可以展开查看其他识别日期并选择 2026-07-20。

#### 场景三：系统没有识别到日期

用户希望将一篇没有明确日期的内容安排到以后阅读。

用户点击「添加到日历」，日期字段为空，由用户手动选择日期。

#### 场景四：用户在 Calendar 页面手动创建事项

用户在 Calendar 页面点击某一天，创建一个不关联知识内容的日历事项。

#### 场景五：查看日历事项的来源

用户在 Calendar 页面点击一个由知识详情页创建的事项，可以查看关联内容并返回原知识详情页。

## 6. 功能范围

## 6.1 知识详情页

### 6.1.1 添加到日历按钮

在知识详情页顶部操作区加入：

```text
编辑
添加到日历
删除
```

建议按钮使用日历图标与文字：

```text
📅 添加到日历
```

按钮适用于所有知识来源：

- wechat
- web
- github
- gpt_chat
- douyin
- manual

### 6.1.2 按钮状态

未创建关联日历事项时：

```text
添加到日历
```

已经创建关联日历事项时：

```text
已添加 · 2026-08-01
```

点击已添加状态后，打开日历事项编辑弹窗，而不是再次创建重复事项。

### 6.1.3 添加弹窗

点击「添加到日历」后，打开弹窗。

弹窗字段：

| 字段 | 必填 | 默认值 |
|---|---:|---|
| 事项名称 | 是 | 知识标题或识别出的事件名称 |
| 日期 | 是 | 系统推荐日期或空 |
| 推荐日期说明 | 否 | 显示识别依据 |
| 关联内容 | 否 | 当前知识标题，只读 |
| 备注 | 否 | 空 |

弹窗示例：

```text
添加到日历

事项名称
[项目报名截止]

推荐日期
[2026-08-01]

识别依据
“报名将在 8 月 1 日截止”

[选择其他日期]

关联内容
某项目申请指南

[取消] [添加]
```

### 6.1.4 用户修改能力

用户可以：

- 修改事项名称
- 使用推荐日期
- 选择其他日期
- 在多个候选日期之间切换
- 添加备注
- 取消操作

用户手动选择的日期始终高于系统推荐日期。

## 6.2 日期识别

### 6.2.1 识别目标

从以下内容中识别日期：

- 原始正文
- 用户粘贴的文本
- 网页正文
- 视频文字稿
- GPT 对话内容
- GitHub README
- 知识摘要

优先使用原始正文。原始正文不存在时，再使用摘要。

### 6.2.2 技术方案

MVP 使用：

```text
正则表达式提取
+
日期标准化
+
上下文关键词排序
```

不要求每次调用大模型。

后续可以增加 LLM 作为低置信度日期的补充识别方式，但不属于当前 MVP。

### 6.2.3 支持的明确日期格式

至少支持：

```text
2026年8月1日
2026 年 8 月 1 日
2026-08-01
2026/08/01
2026.08.01
8月1日
08月01日
8-1
8/1
```

### 6.2.4 支持的相对日期

至少支持：

```text
今天
明天
后天
本周
下周
下周一
本月底
下个月
下月初
```

相对日期需要根据知识条目的导入日期或发布时间转换。

优先基准：

1. 内容原始发布时间
2. 内容导入时间
3. 当前系统日期

### 6.2.5 支持的模糊日期

可识别但需要降低置信度：

| 原文 | 标准化建议 |
|---|---|
| 8月初 | 8月1日 |
| 8月中旬 | 8月15日 |
| 8月底 | 当月最后一天 |
| 下周 | 下周一 |
| 预计9月发布 | 9月1日 |
| 今年年底 | 12月31日 |

模糊日期在弹窗中必须标注：

```text
该日期由模糊时间推算，请确认。
```

### 6.2.6 年份推断

当正文只出现月和日，例如“8月1日”：

1. 优先使用文章发布时间所在年份。
2. 如果该日期已经早于文章发布时间，推断为下一年。
3. 如果没有文章发布时间，使用当前年份。
4. 如果推断后日期已经过去，可以提示用户检查年份。
5. 不应静默生成明显过期的推荐日期。

示例：

- 当前日期：2026-07-15
- 原文：8月1日
- 推荐：2026-08-01

示例：

- 当前日期：2026-09-01
- 原文：8月1日
- 推荐：2027-08-01
- 置信度：中

### 6.2.7 日期上下文关键词

高优先级关键词：

```text
截止
截至
结束
到期
提交
报名截止
申请截止
最后日期
deadline
due
expires
```

中优先级关键词：

```text
发布
上线
开放
开始
举办
召开
开幕
更新
开源
launch
release
start
```

低优先级关键词：

```text
预计
计划
可能
暂定
大约
左右
或将
```

### 6.2.8 多日期排序规则

同一内容中存在多个日期时，按以下顺序推荐：

1. 与截止、提交、到期等关键词关联的未来日期。
2. 日期范围的结束日期。
3. 与活动、发布、开放等事件关联的未来日期。
4. 明确的完整未来日期。
5. 只有月日的未来日期。
6. 模糊未来日期。
7. 普通正文日期。
8. 过去日期。

过去日期默认不作为推荐日期，但可以保留在候选列表中。

### 6.2.9 日期范围

示例：

```text
报名时间：7月20日至8月1日
```

识别结果：

- 开始日期：2026-07-20
- 结束日期：2026-08-01
- 默认推荐：2026-08-01
- 推荐事件：报名截止

当前 CalendarItem 仍然只保存一个日期。

### 6.2.10 置信度

日期识别结果分为：

| 置信度 | 条件 |
|---|---|
| high | 完整日期，且与明确事件关键词关联 |
| medium | 缺少年份，但可以合理推断 |
| low | 模糊日期、预测日期或上下文不明确 |

弹窗显示方式：

```text
高可信度
可能日期，请确认
模糊日期，请手动确认
```

## 6.3 Calendar 独立页面

### 6.3.1 页面入口

主导航新增：

```text
Calendar
```

建议路由：

```text
/calendar
```

### 6.3.2 页面视图

MVP 支持两种视图：

```text
月视图
列表视图
```

默认进入月视图。

不提供小时级周视图和日视图。

### 6.3.3 月视图

月视图应支持：

- 查看当前月份
- 切换上一个月
- 切换下一个月
- 回到今天
- 显示每天的日历事项
- 点击某个日期创建事项
- 点击事项查看或编辑
- 高亮今天
- 区分有关联来源和无关联来源的事项

单个日期中的事项超过可显示数量时：

```text
+3 更多
```

点击后展示当天完整事项列表。

### 6.3.4 列表视图

列表视图按日期升序显示即将到来的事项：

```text
2026-07-20
项目开放报名

2026-08-01
项目报名截止
来源：某项目申请指南
```

列表至少包含：

- 未来事项
- 已过去事项

可以使用页签或筛选：

```text
即将到来
已过去
全部
```

### 6.3.5 新建事项

用户可以从 Calendar 页面手动新建事项。

字段：

| 字段 | 必填 |
|---|---:|
| 事项名称 | 是 |
| 日期 | 是 |
| 备注 | 否 |
| 关联知识 | 否 |

从某个日期格点击创建时，默认使用该日期。

### 6.3.6 编辑事项

用户可以修改：

- 事项名称
- 日期
- 备注
- 关联知识内容

如果事项由知识详情页创建，修改日期不会修改知识内容中识别出的原始日期。

### 6.3.7 删除事项

删除前需要二次确认：

```text
确定删除该日历事项吗？
```

删除日历事项不会删除原知识内容。

### 6.3.8 返回知识详情页

存在 `sourceId` 的事项应显示：

```text
查看关联内容
```

点击后跳转：

```text
/knowledge/{sourceId}
```

## 7. 核心用户流程

### 7.1 从详情页添加

```text
进入知识详情页
→ 点击“添加到日历”
→ 系统读取候选日期
→ 自动填写推荐日期和事项名称
→ 用户确认或修改
→ 点击“添加”
→ 创建 CalendarItem
→ 按钮变为“已添加 · 日期”
```

### 7.2 系统未识别日期

```text
进入知识详情页
→ 点击“添加到日历”
→ 日期字段为空
→ 用户手动选择日期
→ 点击“添加”
```

### 7.3 修改已添加事项

```text
点击“已添加 · 日期”
→ 打开编辑弹窗
→ 修改标题、日期或备注
→ 保存
```

### 7.4 Calendar 页面手动添加

```text
进入 Calendar
→ 点击某个日期或“新建事项”
→ 填写标题和日期
→ 保存
```


```json

  "recommended": {

    "context": "报名将在8月1日截止"
  },
  "candidates": []
```

### 9.2 创建日历事项

```http
POST /api/calendar
```

请求：

```json
{
  "title": "项目报名截止",
  "date": "2026-08-01",
  "sourceId": "knowledge_001",
  "detectedDateId": "date_001",
  "dateSource": "detected"
```

```http
GET /api/calendar?start=2026-07-01&end=2026-07-31
```

```http
PATCH /api/calendar/:id
```

### 9.5 删除日历事项

```http
DELETE /api/calendar/:id
```

### 9.6 手动触发日期重新识别

```http
POST /api/knowledge/:id/detect-dates
```

适用于正文更新后重新识别日期。

### 10.1 加载状态

- 点击「添加到日历」后，弹窗立即打开。
- 日期识别结果仍在加载时显示骨架或加载提示。
- 不应阻塞用户手动选择日期。

### 10.2 空状态

Calendar 没有事项时显示：

```text
还没有日历事项
```

## 11. 业务规则

1. 日历事项必须具有标题和日期。
2. 日期统一保存为 `YYYY-MM-DD`。
3. 当前版本不保存时区和具体时间。
4. 系统识别出的日期不能未经用户确认自动加入日历。
5. 用户可以忽略推荐日期并自行选择。
6. 删除知识条目时，需要处理关联日历事项。
7. 建议删除知识条目时提示用户选择：
   - 保留日历事项并移除关联
   - 同时删除关联日历事项
8. 删除日历事项不会删除知识条目。
9. 同一个知识条目默认只创建一个主要关联日历事项。
10. 后续可以允许一个知识条目关联多个事项，但 MVP 可先限制为一个。
11. 用户再次点击「添加到日历」时，优先打开已有事项，而不是创建重复事项。
12. 过去日期不得作为默认推荐日期，除非没有任何未来候选日期。
13. 所有日期计算应使用用户本地日期。
14. 日期颗粒度只到日，不显示 `00:00`。

---

## 12. 边界情况

### 12.1 同一文章出现大量历史日期

只将未来日期放入主要候选列表，历史日期折叠显示或忽略。

### 12.2 正文出现版本号

例如：

```text
Python 3.12.1
```

不得识别为日期。

### 12.3 正文出现价格或小数

例如：

```text
8.1 元
```

不得直接识别为 8 月 1 日。

### 12.4 月日缺少年份

必须根据内容发布时间或当前日期推断，并标注置信度。

### 12.5 日期已经过期

可以展示候选，但默认不选中，并显示：

```text
该日期已经过去。
```

### 12.6 日期识别错误

用户可以直接修改日期，不需要先删除识别结果。

### 12.7 文章更新

正文更新后可以重新识别日期，但不得自动覆盖已经创建的日历事项。

### 12.8 删除来源内容

关联来源删除后，日历事项可以保留，但显示：

```text
原关联内容已删除
```

---

## 13. 权限与安全

当前为个人知识库，默认只有当前用户可以查看和编辑自己的日历事项。

需要确保：

- 用户只能访问自己的 CalendarItem。
- 用户只能关联自己的 KnowledgeItem。
- API 更新与删除操作需要校验所属用户。
- 前端传入的 `sourceId` 不可直接信任。
- 正文日期识别不执行正文中的代码或脚本。

---

## 14. 性能要求

1. 已经识别过的日期应缓存或保存在数据库中。
2. 打开添加弹窗时，不应每次重新扫描整篇正文。
3. 内容导入完成后可以同步或异步执行日期识别。
4. 日期识别失败不应影响知识条目保存。
5. 月视图只加载当前可见日期范围附近的事项。
6. Calendar 页面首次可交互时间应与其他普通页面接近。

## 16. MVP 验收标准

### 16.1 详情页

- 所有知识类型详情页均显示「添加到日历」按钮。
- 点击按钮可以打开添加弹窗。
- 有识别日期时自动填写推荐日期。
- 无识别日期时允许用户手动选择。
- 用户可以修改标题和日期。
- 创建成功后按钮显示已添加状态。
- 再次点击时打开已有事项编辑弹窗。
- 不会重复创建相同关联事项。

### 16.2 日期识别

- 能识别常见中文完整日期。
- 能识别常见数字日期格式。
- 能处理缺少年份的月日。
- 能识别常见相对日期。
- 能优先推荐与“截止”等关键词关联的日期。
- 能将日期标准化为 `YYYY-MM-DD`。
- 不会把常见版本号识别为日期。
- 模糊日期会显示低置信度提示。

### 16.3 Calendar 页面

- Calendar 为独立一级页面。
- Dashboard 不显示完整日历。
- 支持月视图。
- 支持列表视图。
- 支持手动新建事项。
- 支持编辑事项。
- 支持删除事项。
- 支持从关联事项返回知识详情页。
- 事项只展示日期，不显示具体时间。

### 16.4 数据

- CalendarItem 可以关联 KnowledgeItem。
- 删除日历事项不会删除知识内容。
- 修改日历事项不会覆盖正文识别结果。
- 日期统一保存为 `YYYY-MM-DD`。
- 用户数据权限隔离正确。
